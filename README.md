# Colloquium DevSpace

Colloquium DevSpace — multi-agent среда разработки с общим чатом, файловым контекстом, проектными индексами и MCP-исполнением команд.

Система ориентирована на реальную инженерную работу: редактирование кода, анализ проекта, запуск инструментов и репликацию ответов между участниками (люди, LLM, системные акторы).

## Что внутри

1. Backend API на FastAPI/Python.
2. Postgres-хранилище состояния (чаты, посты, файлы, индексы, сессии).
3. MCP sandbox для выполнения команд и вспомогательных операций.
4. Веб-интерфейс через frontend + nginx-router.
5. Логирование и диагностические логи по компонентам.

## Ключевые возможности

1. Многопользовательские чаты с привязкой состояния к сессии.
2. Выбор активного проекта на уровне сессии.
3. Индексация файлов и сущностей проекта.
4. Обработка специальных блоков в сообщениях (код, патчи, shell-команды, сканирование проекта).
5. Репликация и маршрутизация сообщений между акторами.
6. Поддержка нескольких LLM-провайдеров, включая OpenRouter.

## Архитектура сервисов

1. colloquium-core: основная бизнес-логика API и оркестрация акторов.
2. postgres: основная база данных.
3. mcp-sandbox: выполнение команд и инструментальные операции.
4. frontend: UI-клиент.
5. nginx-router: внешний вход и маршрутизация.

Основной compose-файл: src/docker-compose.yml.
Локальные переопределения: docker-compose.override.yml.

## Быстрый запуск

1. Подготовить секреты в src/secrets.
2. Поднять стек:

```bash
docker compose up -d --build
```

3. Проверить статус:

```bash
docker compose ps
docker logs --tail 200 colloquium-core
```

4. Открыть UI через nginx-router (порт в compose-конфиге).

## Конфигурация и секреты

1. Для подключения core к Postgres используется PGPASSWORD_FILE с секретом в /run/secrets/cqds_db_password.
2. Не дублировать пароль одновременно в PGPASSWORD и PGPASSWORD_FILE.
3. Для локальных окружений использовать override-файл, не меняя боевые значения в основном compose.

## Операционные заметки

1. Изменения Python-кода в core требуют перезапуска контейнера colloquium-core.
2. Изменения Dockerfile/зависимостей требуют пересборки образа.
3. При росте WARN по отсутствующим ссылкам attached_files рекомендуется отдельная чистка с dry-run и валидацией.

## Структура репозитория

## Ролевое разделение LLM-команды

В директории `docs/` находятся примерные пре-промты для подключения разных LLM-моделей к одному Colloquium-чату в разных ролях:

- `llm_pre_prompt.md` — базовые инструкции для всех участников: формат контекста, работа с @agent, sandwich-формат, блоки команд.
- `llm_pre_prompt-claude4o.md`, `-claude4s.md` — арбитр для критических решений и премиальный ревьюер.
- `llm_pre_prompt-gpt5c.md`, `-gpt5n.md` — агентный кодинг-ассистент и быстрый помощник.
- `llm_pre_prompt-grok4c.md`, `-grok4f.md` — простые задачи и аналитика/code review.
- `llm_pre_prompt-nemotron3s.md` — анализ логов и статистика.

> **Примечание:** это демонстрация концепции ролевого разделения команды. Имена файлов соответствуют конкретным провайдерским моделям, которые использовались в production-настройке. Каждый ролевой файл содержит полные базовые инструкции (протокол @agent, контекст, команды), дополненные специфической ролевой конфигурацией. При подключении собственной модели достаточно взять подходящий ролевой файл за основу пре-промта.

## MCP Tool

В составе проекта есть MCP-мосты к Colloquium API для агентов (Cursor, Copilot) и сервисных сценариев.

1. **Рекомендуется по умолчанию:** `src/mcp-tools/cqds_mcp_mini.py` — компактная презентация инструментов, меньше нагрузка на контекст модели. Документация: `docs/cqds_mcp_mini.md`.
2. **Полный набор:** `src/mcp-tools/cqds_mcp_full.py` — все инструменты по отдельности (хост-процессы, БД, расширенный Docker и т.д.). Документация: `docs/cqds_mcp_full.md`.

Имеет смысл, когда стандартного UI-потока недостаточно и нужен программный доступ к служебным возможностям CQDS.
