# Cached LLM Interact Design

Date: 2026-03-27 (baseline update: 2026-04-08; reference commit spec: 2026-04-06)
Scope: /app (local working tree at P:/opt/docker/cqds)

## Goals

- Reduce repeated full-context sends during interactive chat.
- Keep correctness when context head changes (index/file changes, old post edits, hidden legacy posts).
- Support majority-path updates where 9/10 changes are tail appends.
- Keep fallback path: full resend of pre-prompt + index + sandwiches whenever cache assumptions break.

## Быстрое наполнение Phase 1 без вызова LLM

Для **первичной** статистики по сборке контекста и решению кэша живой ответ провайдера **не обязателен**: достаточно `build_context` и записи в `context_cache_metrics`.

Если **триггерный** пост (тот, с которого идёт `_broadcast`) содержит подстроку **`#debug_bypass`** (регистр не важен), `interact` после сборки контекста **не вызывает** провайдера и возвращает заглушку; в метриках `error_type` будет **`debug_bypass`** (при глобальном серверном `debug_mode` — **`debug_mode`**). В сценариях: `mcp-tools/scripts/filewalk_review_runner.py --debug-bypass` (к таймауту ожидания «ответа» можно добавить `--wait-timeout 20`).

Так не измеряются задержки и токены **completion** у провайдера, и в debug-ветке не нужно трактовать `used_tokens`/`output_tokens` как экономику. Этот режим предназначен для быстрой проверки алгоритма (`mode`/`reason`, `build_context_ms`, структура контекста) и дешёвого накопления статистики решений.

Отдельно для диагностики инкрементальных правок может писаться доля апгрейда контекста (`context_upgrade_tokens`, `context_full_tokens`, `context_upgrade_pct`) — это структурная метрика промпта, а не биллинг провайдера.

Готовая **лесенка** прогонов (растущий объём, разные `files_per_chat` и `offset`): `mcp-tools/scripts/debug_stats_ramp.py` (все шаги или `--step-begin` / `--step-end`). Отчёты: `mcp-tools/logs/cache_phase1_runner/debug_ramp/`.

Для проверки **DELTA_SAFE** (текстовый чат без вложений): `mcp-tools/scripts/delta_safe_chat_runner.py` — один чат, N постов с `#debug_bypass`.

## Current Pipeline (as implemented)

### Request and replication chain

1. POST message enters chat route and lands in PostManager.add_post/process_post.
2. process_post may trigger replication via ReplicationManager.check_start_replication.
3. ReplicationManager.replicate_to_llm calls _broadcast.
4. _broadcast builds content blocks via collect_blocks.
5. collect_blocks order:
   - assemble_posts(chat)
   - assemble_files(attached_files)
   - assemble_spans()
6. For each target actor, _recursive_replicate calls interact(ci, rql).
7. interact builds context in build_context and sends single prompt through llm_connection.make_payload(prompt=context).

### Current ordering behavior and implications

Posts:
- scan_history loads posts ordered by p.id ASC (oldest -> newest).
- assemble_posts iterates reversed(history.values()) (newest -> oldest).
- Relevance is reduced by count while iterating, so newest posts keep highest effective priority.

Files and spans:
- Files are appended after posts.
- Spans are appended after files.

Practical effect:
- The assembled block sequence is currently: latest posts first, then file blocks, then span blocks.
- This matches the historical intent of focusing models on fresh conversational tail before full file payload.

### Token limit in build_context (trimming)

**Status:** В цикле `build_context` при принятии блока выполняется **`total_tokens += block_tokens`** — накопление учитывается, обрезка по лимиту идёт по суммарному весу (см. `agent/llm_interactor.py`).

## Why cache must be hybrid (incremental + full fallback)

Context head mutability events:
- File index changed.
- Early history edited/deleted.
- Legacy visibility policy changed (hide old posts on demand).
- Pre-prompt changed.

These events invalidate assumptions about immutable prefix.
For these cases, patching old sent body is unsafe and a full resend is required.

Tail-dominant events:
- New post appended.
- New mention/attach in latest exchange.

These can be handled with delta-safe flow if signatures match.

## Proposed Cache State

Key:
- actor_id
- chat_id
- session_id
- provider/model
- pre_prompt_hash

State object (ContextCacheState):
- provider_session_id (optional, if provider supports session/prompt-cache)
- mode: FULL or DELTA_SAFE
- last_post_id
- pre_prompt_hash
- index_hash
- head_hash
- tail_hash
- last_file_rev
- last_span_rev
- updated_at
- expires_at

Storage:
- Primary: process memory registry + named lock + TTL/LRU.
- Optional persistence: sqlite metadata table for restart diagnostics.

## Invalidation Rules (force FULL)

- pre_prompt_hash changed
- index_hash changed
- edit/delete targets post_id <= cached last_post_id
- visibility policy changed for old messages
- model/provider/reasoning config changed
- cache TTL expired
- provider session rejected or diverged

### Pitfalls: `index_hash` и метаданные пака

- `index_hash` считается по полному JSON индекса из `SandwichPack.pack`, поэтому **любое** нестабильное поле в том же объёме (не только список файлов) меняет хэш на каждом вызове и даёт ложные срабатывания вроде «индекс изменился».
- Пример: `context_date` с точностью до секунды делало хэш уникальным при каждой сборке пака. Решение — оставить в индексе только **дату в UTC** (`YYYY-MM-DD`): внутри календарных суток UTC поле стабильно; раз в сутки по UTC — ожидаемый сдвиг (для гибридного кэша обычно терпимо).
- **Подводный камень:** граница «дня» — по UTC, не по локальному часовому поясу пользователя или хоста.
- Новые поля в тот же блоб добавлять с осторожностью: они должны зависеть от модели индекса/файлов, а не от `now()` с субсекундной или секундной гранулярностью, если эти поля участвуют в сериализации для хэша.

Checklist:

**Замечание про инвалидацию и префикс:** цепочка инвалидации (FULL, сброс **B**, смена окна) в типичном продукте делает **накопленный материализованный префикс** ненужным в **очень большой доле** случаев — это не баг, а следствие частых пересборок окна; сценарии ниже нужны там, где хочется **не FULL**, а явную семантику «пост исчез из видимого окна, но модель должна забыть старое тело».

### Cutout: удалённые посты, ещё «живые» в памяти модели

**Проблема:** часть постов **исчезает** из активного окна (модерация, скрытие, жёсткое удаление из истории, сдвиг окна так, что пост больше не попадает в `filtered_blocks`), тогда как в **предыдущем** промпте этот `post_id` ещё был. Без явного сигнала модель может опираться на устаревший текст так же, как при тихой правке.

**Консервативный путь (уже близок к текущим правилам):** любое изменение набора идентичностей постов в «голове» до `last_post_id` ломает `head_posts_sig` → **`FULL`** / сброс **B**; mainline пересобирается без удалённого поста. Минус — чаще полная пересборка и потеря DELTA_SAFE-цепочки.

**Инкрементальный путь (предложение):** трактовать удаление как **редакцию содержимого** по тому же `post_id` и выражать её **хвостовым** `:context_patch` (или эквивалентом с тем же приоритетом, что у `context_revision`):

- в теле патча — компактный **tombstone**, например строка вида **`#deleted_by:<user_or_system>`** (или канонический `__DELETED__` + атрибуты `deleted_by`, `deleted_at` в разметке блока, если так удобнее парсеру);
- в **`post_ref` / `post_digest`** после commit хранить дайджест именно этого tombstone, чтобы на следующем ходе не дублировать патч;
- в **препромпте** явно: «если для `post_id` пришёл патч с пометкой удаления, считать прежний текст поста **недействительным** даже если фрагмент ещё виден в старом префиксе у провайдера».

#### Двухфазное удаление в БД и сборщик мусора

Целевая **операционная** модель (согласуется с tombstone `#deleted_by:<actor>`):

**Фаза 1 — пока есть «живые» ссылки на пост как на удаляемый объект**

- В БД тело поста **не стирается физически сразу**: контент **заменяется** на канонический tombstone, например **`#deleted_by:<actor>`** (идентичность `post_id` и строки в истории чата сохраняются для стабильности ссылок).
- **Бэкенд** в ответах API для фронта такие посты **не включает** — для пользователя это **эффект полного удаления** из чата (фронт ничего не «вырезает» сам; отсутствие строки — контракт API).
- **Сборка контекста для FULL** (и в целом «чистое» окно для модели): те же правила на стороне ядра — логически удалённые посты **не попадают** в поток блоков для LLM (исключение на этапе `assemble_posts` / общего слоя доступа к истории), либо в промпт попадает только узкая заглушка по политике cutout. Инкрементальные пути при необходимости всё ещё видят **тот же `post_id`** в `post_digest` / патчах, пока кэш не забыл строку.

**TODO (на будущее) — общая «Корзина» в БД**

- Реализовать **единую корзину** удалённых постов **на все чаты** (отдельная сущность/таблица или раздел в схеме), чтобы пользователь мог отменить **ошибочное** удаление в разумный срок.
- Пока корзина не сделана, поведение «как удалили навсегда» на UI остаётся, но фаза 1 в БД всё равно даёт окно для кэшей и для будущего UX восстановления.
- **Восстановление поста из корзины** в активный чат подразумевает **обязательную FULL-интеракцию** (и сброс материализованного префикса **B**, инвалидация доверенного префикса для delta): состав окна и дайджесты меняются нетривиально, цепочку DELTA_SAFE продолжать без полной пересборки нельзя.

Под **«ссылками»** имеются в виду любые удерживающие структуры: записи в **`ContextReferenceStore`** (`post_digest` и т.д.), материализованный префикс **B**, провайдерский session/prompt-cache (Phase 3), явные **ref-count** в БД по чатам/сессиям, если введёте — до обнуления всех таких ссылок физическое удаление строки **запрещено**.

**Фаза 2 — после исчезновения ссылок**

- Фоновый **сборщик мусора** удаляет строку поста из БД **полностью** (и при необходимости связанные объекты), когда для данного `post_id` не осталось ни одной учётной ссылки из списка выше.
- Запуск GC **при старте ядра** — разумная страховка: подчистить «осиротевшие» tombstone-записи после аварийного завершения, истечения TTL сессий и т.п.
- После фазы 2 `post_id` исчезает из мира — любые старые логи/дампы с этим id трактуются как исторические; новые кэши его не ждут.

**Следствия для кэша:** пока фаза 1 активна, дайджест в **A** для этого id стабильно соответствует tombstone; патчи «снятия» контента **дублировать** не нужно, если FULL-сборка уже не подмешивает старый текст. Переход в фазу 2 снимает id из `post_digest` естественным образом (запись в БД нет → нет блока); дифф «id исчез» обрабатывается правилами cutout / `head_posts_sig` или отдельным проходом GC по снимкам.

**Варианты и ограничения:**

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| Только FULL при удалении из окна | Простота, один источник истины в промпте | Дороже по сборке/транспорту; чаще рвётся цепочка |
| Tombstone в `context_patch` | Сохраняется инкрементальная семантика для модели и для будущего delta-транспорта | Нужны правила в препромпте, учёт в `append_incremental_patches`, метрика `reason` (например `post_deleted_tombstone`); при **полном** исчезновении `post_id` из потока блоков нужно **явно знать** список «был в prev, нет в текущем mainline» |
| Скрытие без удаления id (пост остаётся в БД как заглушка) | Упрощает дифф «тот же id — новое тело» | Модель всё ещё видит строку в mainline; UX политики скрытия |

**Связь с `head_posts_sig`:** если удалённый пост **пропадает** из канонического списка постов до `p_prev`, подпись головы **меняется** — при нынешнем `_decide_cache_mode` это уже **FULL** (`head_posts_changed`). Чтобы остаться в инкрементальной ветке, пришлось бы либо **расширить** модель подписи (например, отдельный набор «присутствующих id» vs «tombstone-only id»), либо **оставлять** пост в потоке как блок-заглушку с tombstone в mainline (спорно для UX). На практике разумна **ступенчатая** политика: редкие удаления → FULL; частые или требования к провайдерскому кэшу → доработка патчей и контракта.

## Delta-safe Rules

Allow DELTA_SAFE only when all are true:
- New post id strictly greater than cached last_post_id.
- No index/file/spans revision rollback.
- No pre-prompt change.
- No old-history mutation marker.

### Реализация (LLMInteractor/LLMCachedInteractor, апр. 2026)

Слой интерактора разделён по наследованию:

- `LLMInteractor` — базовый транспортный слой.
- `LLMCachedInteractor` — промежуточный слой кэш-политики контекста.
- `ReplicationManager` наследуется от `LLMCachedInteractor`.

Быстрый rollback для отладки: в `agent/managers/replication.py` вернуть базовый класс
`ReplicationManager` c `LLMCachedInteractor` обратно на `LLMInteractor`.

Разделение **«закэшированный префикс»** vs **инкремент (новые посты)** в решении `ContextCacheDecision`:

- **`head_posts_sig`** — SHA-256 от канонического списка **идентичностей** постов с `post_id <= last_post_id` (в строку входят `post_id` и `user_id`, **без** текста). Иначе каждый цикл «⏳ progress → финал» под тем же id ломал бы сравнение и `DELTA_SAFE` не наступал бы.
- **`post_digest`** (снимок в `context_fp_cache`) — карта `post_id` → дайджест текста; для прогресс-строк используются псевдо-токены `__PROGRESS__` / `__WARN__`. Для постов с id ≤ предыдущего `last_post_id` смена дайджеста (кроме перехода `__PROGRESS__` → финальный текст) даёт `FULL` / **`head_post_content_changed`**.
- **`non_post_sig`** — отдельная подпись файлов и spans (сортировка по строке идентичности). Блоки **`:context_patch`** в эту подпись **не** входят. Любое изменение вложений/спанов между ходами → `FULL` / `attachments_or_spans_changed`.
- **`DELTA_SAFE`** выставляется только если совпали `head_posts_sig` и дайджесты «головы» по `post_digest`, `non_post_sig`, индекс и pre-prompt не менялись, а текущий `last_post_id` **строго больше** прежнего (чистое наращивание хвоста по id).

### Cache hit / cache miss (формализация)

В Phase 1 транспорт **всё равно шлёт полный собранный контекст**; поля `mode` / `reason` в `context_cache_metrics` — это **логическое** решение «можно ли считать префикс неизменным для *будущего* инкрементального транспорта или prompt-cache у провайдера», а не факт урезанного HTTP-тела.

| Итог в метриках | Смысл | Когда |
|-----------------|--------|--------|
| **`DELTA_SAFE` (cache-hit)** | Префикс по правилам ниже **стабилен**; новый материал — только **хвост** (`last_post_id` вырос, «голова» по id и по дайджестам совпала, индекс не-постов и pre-prompt те же). | Строго монотонное добавление **нового** поста без правок в уже зафиксированной «голове». |
| **`FULL` (cache-miss)** | Любое другое условие: нет снимка, смена pre-prompt / индекса / подписей / **текста поста в голове** / состава не-постов; нет роста `last_post_id`; откат id; смена окна в `filtered_blocks`, дающая другой набор постов в подписи «головы». | См. колонку `reason` в той же строке метрик. |

**Важно:** *инкрементальное редактирование для модели* (`:context_patch` в хвосте промпта, см. ниже) и *cache-hit в таблице метрик* — **разные оси**. Патч добавляется, когда текущая сборка **расходится** с **предыдущим** снимком по тексту поста или `revision_ts` файла — часто в том же запросе, где решение уже **`FULL`** (например, после правки старого поста и нового сообщения в хвосте: **`head_post_content_changed`**). То есть:

- **Метрика `FULL`** = «мы **не** утверждаем, что префикс как у прошлого вызова остался доверенным для delta-only отправки».
- **Блок `<context_patch>`** = «в **этом же** полном промпте явно продублирована **актуальная** версия поста/файла, чтобы модель (и в перспективе провайдер с кэшем префикса) не опиралась на устаревший фрагмент».

Иными словами: правка поста с `id ≤` прошлого `last_post_id` **ломает cache-hit** (получите **`head_post_content_changed`** при следующем вызове LLM, если к тому моменту уже был снимок с `post_digest`). Это **ожидаемо** для *текущего* кода. Инкремент здесь — не «остались в `DELTA_SAFE`», а «в промпт **добавлен** хвостовой патч + в препромпте правило приоритета»; полный пересбор контекста при `FULL` по-прежнему включает актуальные `<post>` из БД — патч **дополнительно** помечает пересмотр для сценария дубликатов id.

### Целевая модель: заморозка референса и patch-first (пока не сведена с кодом)

Задумка **шире**, чем то, что сейчас делает `ContextCacheDecision`:

1. **Референс-заморозка** — посты и файлы, уже попавшие в «префикс» соглашения, трактуются как **стабильные ревизии** в данных отсчёта кэша: не сравнивать целиком «новый хэш всего старого тела» с прошлым снимком как триггер miss.
2. **Патчи из диффа** — любая актуализация старого `post_id` / `file_id` выражается **хвостовыми** `context_patch` (и при необходимости явными rev-id), а не инвалидацией всего префикса.
3. **Основные причины cache-miss** — смена **pre-prompt**, смена **индекса** (не-постовая модель чата/проекта), **структурные** события (удаление поста, откат id, несовместимая с контрактом смена окна релевантности), плюс явный «сброс по политике», а **не** «постарел текст поста, формально заменённый ревизией в хвосте».

**Разрыв с реализацией (апр. 2026):** сравнение **`post_digest`** по «голове» и жёсткий **`head_post_content_changed` → `FULL`** — **консервативный** путь: любая правка текста в уже учтённых id даёт miss. Это **не** реализует описанную freeze+patch-first семантику; патчи в промпте есть, но **метрика** всё равно уходит в miss.

**Направления доработки кода** (черновик для будущего шага):

- Ввести отдельный режим решения (или флаг): при **неизменных** `pre_prompt_hash`, `index_hash`, `head_posts_sig` (id-набор до `p_prev`) и **строгом** росте `last_post_id` не переводить в `FULL` только из‑за смены дайджеста старого поста, **если** эта смена **покрыта** сгенерированными `context_patch` (и при необходимости ограничить класс правок, e.g. только редактирование без удаления).
- Для файлов — аналог: не опираться на полное совпадение **`non_post_sig`** по содержимому как единственный критерий, если рост `revision_ts` отражается патчем; miss оставить для появления **нового** `file_id` в индексе / смены набора вложений.
- Зафиксировать в метриках новый **`reason`** вроде `tail_append_with_patches` vs старый `head_post_content_changed`, чтобы различать поколения логики.

Пока этого нет, таблица cache-hit/miss выше описывает **фактическое** поведение агента; абзац про целевую модель — **спецификация наперёд**.

### Референс «что уже скормили модели»: структура и обновление (целевая спецификация)

**Задача:** хранить не «состояние чата в БД», а **снимок того, что реально ушло в последний исходящий промпт** (после токен-обрезки и **включая** хвостовые `context_patch`). От него считается дифф на следующей интеракции: иначе одни и те же правки будут снова и снова порождать патчи или ложно считаться «новыми» постами.

#### Почему не один последовательный список хэшей

Плоский список хэшей **без ключей** не подходит как основная структура: нужно **адресно** обновлять ревизию по `post_id` / `file_id` после каждого патча. Достаточно:

| Слой | Назначение |
|------|------------|
| **Словари по id** | `post_ref: post_id → запись`, `file_ref: file_id → запись` — быстрый дифф «БД/сборка vs то, что модель уже видела». |
| **Опционально `interaction_seq`** | Монотонный счётчик успешных сборок контекста для ключа `(actor_id, chat_id, …)` — отладка, трассировка, будущий аудит. |
| **Опционально журнал событий** | Append-only список `(seq, kind, id, …)` *поверх* словарей — если нужна воспроизводимость; **материализованное состояние** для алгоритма всё равно словари + глобальные хэши. |

Последовательность **порядка блоков** в последнем промпте можно не хранить целиком, если не строите от неё решений; для отладки полезен `order_snapshot: post_id[]` — порядок **первого** появления в mainline (не обязателен для MVP).

#### Запись в `post_ref` / `file_ref` (минимальный контракт)

Рекомендуемая запись (логически):

- **Пост:** `post_id` → `{ prompt_digest, source_ts? }`, где `prompt_digest` — детерминированный хэш **текста в том виде, как он сериализован в сэндвич** (после `filter_input` / нормализации, согласованной с диффом). Псевдо-токены вроде `__PROGRESS__` для прогресс-строк — по тем же правилам, что и сейчас для `post_digest`.
- **Файл:** `file_id` → `{ revision_ts, optional content_digest }` — то, что ушло в последний промпт (для файла обычно достаточно монотонного `revision_ts` из БД, если контент однозначно определяется им).

Плюс на уровне снимка (как уже есть): `pre_prompt_hash`, `index_hash`, `last_sent_max_post_id` (максимум `post_id` среди постов, **фактически присутствующих** в последнем промпте — mainline ∪ посты, затронутые только патчем, по политике ниже).

#### Commit после каждой интеракции (обязательный порядок)

Выполнять **после** финализации списка блоков, уходящих в `pack` (включая патчи):

1. **Собрать кандидат-референс** `ref'` с нуля или копированием предыдущего — см. политику trim ниже.
2. Пройти **финальный** упорядоченный список блоков промпта (как в `filtered_blocks` после `_append_incremental_patches`):
   - для каждого `<post>` (mainline): `post_ref'[id] = digest(тело блока)`;
   - для каждого `<context_patch kind=post>`: **`post_ref'[id] = digest(тело патча)`** — перезапись; для модели авторитетна **последняя** копия в потоке.
3. Файлы: аналогично mainline-блоки и `kind=file` патчи обновляют `file_ref'`.
4. Выставить `last_sent_max_post_id = max` по всем `post_id`, которые **есть в этом промпте** (mainline или post-патч).
5. Атомарно заменить сохранённый референс для ключа кэша на `ref'` **вместе** с `pre_prompt_hash` / `index_hash` этой сборки.

**Инвариант:** если на шаге *N* для `post_id = p` в промпт вошёл патч с телом *T*, то после commit `post_ref[p].prompt_digest = hash(T)`; на шаге *N+1* патч для того же *p* **не генерируется**, пока БД/сборка не даст новое расхождение с этим дайджестом.

#### Политика при токен-обрезке (что считать «скормили»)

Если пост **есть в чате**, но **не вошёл** в промпт из-за лимита токенов:

- **Рекомендуемая политика A (строгая):** такой `post_id` **не входит** в `post_ref'` для этой интеракции (или явно помечается `absent_from_prompt`). Тогда референс отражает **только видимое модели**. При следующей сборке появление поста в окне — легитимный «новый» вклад в контекст, а не «дубль каждый раз».
- **Политика B (мягкая, сложнее):** хранить «последнее когда-либо отправленное» даже если в этом ходе пост выпал — риск рассинхрона с prompt-cache провайдера; нужны отдельные правила.

Для целевой freeze+patch-first ориентир — **A**, пока явно не доказана необходимость B.

#### Связь с cache-hit / patch-first

- Дифф для патчей: `hash(сборка из БД для id)` сравнить с `post_ref[id].prompt_digest` (и аналог для файлов).
- Решение **miss** по препромпту/индексу — не трогает словари, пока не сменились эти глобальные поля; при miss полный сброс или помеченная «эпоха» референса — отдельное решение (проще **полный сброс** `post_ref`/`file_ref` при `pre_prompt_changed` / `index_changed`).

#### Краткий ответ на «список или сложнее»

**База:** два словаря по id + глобальные хэши + `last_sent_max_post_id` + опционально `interaction_seq`.  
**Сложнее — по необходимости:** журнал событий и снимок порядка блоков для аудита, не как единственный источник истины для диффа.

Статистика **`context_cache_metrics` до `schema_ver=2`** собиралась со старой логикой (единый `blocks_signature` по всему списку блоков), из-за чего **`DELTA_SAFE` фактически не достигался**; строки с `schema_ver=1` лучше не смешивать с новыми при оценке доли инкрементальных режимов.

**`schema_ver=3`:** поле `index_hash` в метриках считается по индексу **только не-постовых** блоков (`:post` исключены). Раньше использовался полный `pack(all blocks)` — каждый новый пост менял JSON индекса и давал ложный `index_changed`, что делало `DELTA_SAFE` невозможным даже в чисто текстовом чате.

**`schema_ver=4`:** то же поколение логики с id-only `head_posts_sig` (метрики отличаются от `schema_ver=3` по смыслу поля «голова» постов).

**`schema_ver=5`:** в снимок кэша добавлены `post_digest`, `file_rev_ts`; в конец контекста могут добавляться блоки **`:context_patch`** (см. подраздел ниже). Пока в памяти нет ключа `post_digest`, один раз срабатывает **`stale_fingerprint_layout`** (миграция формата).

### Модуль референс-хранилища (`agent/lib/context_reference_store.py`)

- Класс **`ContextReferenceStore`** держит снимки fingerprint в памяти процесса по ключу `actor_id:chat_id:session_id` (как раньше `context_fp_cache`).
- Переменная окружения **`CQDS_CONTEXT_REFERENCE_STORE`**: значения **`0`**, **`false`**, **`off`**, **`no`** (регистр не важен) — **отключить** расширенный референс: не сохраняются и не используются `post_digest` / `file_rev_ts`, не вызывается генерация **`context_patch`**, проверка **`head_post_content_changed`** и требование ключа `post_digest` в снимке **не** применяются; в хранилище остаётся **узкий** набор полей (`pre_prompt_hash`, `index_hash`, `head_posts_sig`, `non_post_sig`, `last_post_id`, …). В **`context_cache_metrics`** при отключённом референсе пишется **`schema_ver=4`**, при включённом — **`5`**.
- В **`LLMInteractor(..., context_reference_store=...)`** можно передать свой экземпляр (например, с `enabled=False`) для тестов без env.

Вспомогательные функции модуля: **`post_digest_map`**, **`file_rev_ts_map`**, **`append_incremental_patches`**, **`map_get`**, **`post_digest_for_cache`**.

### Инкрементальные правки в контексте (`:context_patch`, schema_ver 5)

После основного прохода по блокам `build_context` в хвост списка могут добавляться **ContextPatchBlock** (`:context_patch`), в сэндвиче это теги вида `<context_patch kind="post|file" role="context_revision" post_id="…" file_id="…" mod_time="…" revision_ts="…">…</context_patch>`.

Порядок в одном запросе: сначала основной список блоков (и обрезка по токенам), затем **`append_incremental_patches`** из `context_reference_store` по **предыдущему** снимку из **`ContextReferenceStore`**, затем **`_log_context_fingerprint`** и запись **`mode`/`reason`**. Поэтому при правке «головы» возможна комбинация **в промпте есть `context_patch`**, а в метрике — **`FULL` / `head_post_content_changed`**: патч про **семантику для модели**, `FULL` про **недопустимость delta-only префикса** (при отключённом референсе патчи не добавляются).

- **Пост:** если с прошлого LLM-вызова изменился дайджест текста поста (и это не только «снятие» псевдо-прогресса), в хвост кладётся патч с актуальным телом.
- **Файл:** если вырос `revision_ts` вложения относительно `file_rev_ts` в снимке, в хвост кладётся патч с актуальным содержимым; пакер не дублирует разбор сущностей файла (блок обрабатывается как пост по пайплайну).

У обычных `<post>` в разметке при сборке указываются **`mod_time`** и при необходимости **`revision_ts`** (Unix), чтобы модель могла сопоставлять версии.

Контракт для модели закреплён в **`docs/llm_pre_prompt*.md`**: при одном и том же `post_id` / `file_id` **авторитетна** копия в `context_patch` и/или с **большим** `revision_ts` / более новым `mod_time`.

### Семантика FULL, холодное ядро и материализованный префикс

**FULL** в смысле кэша сборки означает: **пересчитать всё**, что попадает в текущее **контекстное окно** (релевантность постов, лимит токенов, состав файлов/spans) — без опоры на сохранённый **материализованный префикс** слоя **B**. В промпт уходит целостное тело, соответствующее **актуальному** окну после этой пересборки (плюс при необходимости хвостовые `:context_patch` по правилам Phase 1).

**Холодное ядро, уже существующий чат:** для ключа `(actor, chat, session)` в памяти **ещё нет** снимка — формально контекстный кэш пуст, первый вызов даёт **`no_cache_state` → FULL**. Добавляется новый пост (или триггер репликации), собирается полный контекст по текущему окну и уходит провайдеру. Это **тот же класс интеракции**, что и явный FULL после инвалидации: нет переиспользуемого префикса **B**, только (после `put`) нарастающий слой **A** для диффов и решений на следующих ходах.

**Почему при FULL обязателен сброс префикса (B):** при **сжатии** контекста (токен-лимит) и **скользящем окне** релевантности состав «головы» между ходами меняется с **большой вероятностью**. Старый материализованный префикс почти наверняка перестаёт соответствовать новому набору постов в окне; держать его в памяти бессмысленно и рискованно. Поэтому инвалидация **B** при FULL — не отдельная эвристика, а следствие семантики «полная пересборка окна».

### Слои состояния (раздельно по назначению)

| Слой | Ключ | Что хранит | Жизненный цикл |
|------|------|------------|----------------|
| **A. Решение кэша (telemetry + правила)** | `actor_id:chat_id:session_id` | Узкий fingerprint + при включённом референсе `post_digest`, `file_rev_ts`, сигнатуры головы/не-постов | Процесс ядра; уже в `ContextReferenceStore` |
| **B. Материализованный префикс (новое в Phase 2)** | тот же ключ | Необязательно: последний **сериализованный** префикс промпта (строка или список сэндвич-блоков **до** хвоста новых постов), граница `last_materialized_post_id`, версия хэшей, с которых префикс построен | Только пока допустим **DELTA_SAFE**-путь сборки; см. политику eviction ниже |
| **C. Якорь окна релевантности** | `chat_id` + `session_id` | `last_post_id` на момент последнего FULL | Уже в `relevance_window_anchor` |
| **D. (опционально Phase 3)** | provider session id | Кэш префикса у провайдера | Вне процесса / по API провайдера |

Слой **A** нужен для `_decide_cache_mode`, патчей и метрик. Слой **B** — единственный «тяжёлый» кэш в смысле дублирования **текста** префикса в памяти; без него Phase 2 может ограничиться логикой «не пересобирать лишние блоки» без хранения полной строки (микро-оптимизация) — но полноценная экономия CPU/IO на больших чатах обычно предполагает **B**.

### Порядок одной интеракции (логический pipeline)

1. **Вход `build_context`:** загрузить снимок **A** (`prev`). При наличии **B** — загрузить материализованный префикс только если намерение — попытка delta-сборки (см. eviction «перед FULL»).
2. **Сборка блоков** из БД/ассемблера (как сейчас).
3. **Патчи:** `append_incremental_patches(..., prev, ...)` — требует из **A** полей `post_digest` / `file_rev_ts` у **предыдущего** успешного снимка; выполняется **до** финального решения mode/reason в текущем коде.
4. **Решение:** `_decide_cache_mode(prev, ...)` → `FULL` или `DELTA_SAFE`.
5. **Сборка текста контекста:** при Phase 2 и `DELTA_SAFE` — при наличии **B** и совпадении хэшей с `prev` склеить `prefix_cached + delta_body`; иначе полная сборка (как сейчас).
6. **Commit:** записать новый снимок **A** (`put`). Обновить или создать **B** только если итоговый режим **`DELTA_SAFE`** и политика хранения префикса включена.
7. **`interact`:** отправка провайдеру (Phase 2: по-прежнему полный `context` в HTTP, если нет Phase 3).

### Освобождение памяти и инвалидация «перед каждой FULL интеракцией»

Формулировка **«освободить контекст кэша перед каждой FULL интеракцией»** относится к слою **B (материализованный префикс)** и к любым **временным буферам** полной копии предыдущего промпта, а не к удалению **A** до шагов 3–4 текущего хода (иначе ломаются патчи и решение).

**Инварианты:**

1. **Не** удалять запись **A** для ключа до завершения шагов 3–4 в том же `build_context`, если нужны `post_digest` / сигнатуры для сравнения с `prev`.
2. **После** того как для текущего хода зафиксировано решение **`FULL`** (в конце `build_context`, сразу после `_decide_cache_mode` / эквивалента):
   - **обязательно** сбросить слой **B** для этого `cache_key` (`evict_materialized_prefix(cache_key)`), если он был;
   - не держать вторую копию полного предыдущего промпта «на будущее» — следующий успешный путь снова нарастит **B** только после коммита **DELTA_SAFE**.
3. **Перед началом следующей** интеракции для того же ключа: если в **A** в прошлом коммите сохранён маркер `last_committed_mode=FULL` (или эквивалент), слой **B** уже должен быть пуст (п. 2); дополнительная защита — **идемпотентный** `evict_materialized_prefix` в начале `build_context` при `prev.last_committed_mode == FULL` или при любом заранее известном форс-FULL (например, глобальный kill-switch кэша).
4. Замена **A** через `put` уже отпускает предыдущий dict для GC; при росте числа чатов добавить **LRU/TTL по ключам** для всего `ContextReferenceStore` (отдельный чек-лист), чтобы не раздувать процесс при долгой работе ядра.

**Сводка:** «Перед FULL» на практике реализуется как **удаление материализованного префикса (B)** при фиксации FULL в этом же ходе плюс **входная зачистка B** на следующем ходу, если прошлый коммит был FULL. Это согласовано с тем, что FULL = полный пересчёт окна (как при холодном старте по ключу), а скользящее окно и тримминг делают повторное использование старого **B** малополезным. Снимок **A** (хэши и карты дайджестов для следующего диффа) **сохраняется** и обновляется `put`-ом — иначе нечему сравнивать хвост на следующем шаге.

### Согласование с токен-тримом и `post_ref` (документ выше)

- Политика **A (строгая)** для «что модель видела»: в **A** после commit хранить дайджесты только для постов, **вошедших** в отфильтрованный промпт (и патчи). При Phase 2 слой **B** должен соответствовать тому же префиксу/хвосту, что ушли в строку, иначе повторное использование даст рассинхрон.
- При обрезке по лимиту и сдвиге окна релевантности FULL будет частым исходом; **B** большую часть времени пуст или короткоживущий — это **нормальная** эксплуатация, а не провал кэша.

### Чек-лист имплементации Phase 2 (код)

- [ ] Ввести слой **B** (или явный no-op режим с флагом): API `get/set/evict_materialized_prefix(cache_key)`.
- [ ] При записи метрик/commit **A** сохранять `last_committed_mode` ∈ {`FULL`,`DELTA_SAFE`} для входной зачистки **B** на следующем ходе.
- [ ] В точке фиксации **`FULL`** текущего хода вызывать `evict_materialized_prefix` (после патчей и решения, до/после `put` — без разницы для **B**, главное — до удержания пика памяти между интеракциями не копировать префикс зря).
- [ ] Сборка `DELTA_SAFE`: условие использования **B** — совпадение `pre_prompt_hash`, `index_hash`, сигнатур с теми, на которых префикс построен; иначе `evict` + полная сборка.
- [ ] (Опционально) LRU/TTL по ключам **A** для ограничения памяти при большом числе чатов.
- [ ] Тесты: после FULL следующий ход не использует устаревший префикс; при DELTA_SAFE префикс совпадает с прошлой отправкой.

## Correct Path Before Cache Rollout

This section defines the safer order of work before enabling any transport/context reuse.

Principle:
- Do not optimize cache path while base context assembly is unstable.
- First stabilize token trimming and measurement, then trust cache-mode telemetry.

Step 0 (mandatory fix-first):
- [x] Fix token accumulation in `build_context` filtered-block loop (`total_tokens += block_tokens` when block accepted) — сделано в `agent/llm_interactor.py`.
- [ ] Add regression tests for deterministic trim order under token pressure.
- [ ] Re-check that newest posts still win and fresh files are appended only when room remains (ручная/авто верификация под лимит).

Step 1 (phase-1 telemetry hardening):
- Keep behavior FULL-send (no payload delta transport changes).
- Record decision and reason for each call in machine-readable events:
  - mode: FULL|DELTA_SAFE
  - reason: no_cache_state|stale_fingerprint_layout|pre_prompt_changed|index_changed|head_posts_changed|head_post_content_changed|attachments_or_spans_changed|history_rewind_or_edit|no_tail_append|tail_append_detected|rollout_disabled|cache_internal_error|…(forced invalidate reasons)
  - actor_id, chat_id, session_id, project_id, model, rql
  - build_context_ms, blocks_count, sent_tokens, used_tokens (if available)
- Add explicit version tag in event schema (`cache_telemetry_schema=1`) to support future migration.

Step 2 (phase-1 acceptance gate):
- Run telemetry-only for at least 24h (prefer 48h on mixed chats).
- If there is **no organic usage** yet, use **synthetic scenarios** over multiple days instead; same required outputs below, but interpret them as a **dev baseline** (see «When the product is dev-only»).
- Required outputs:
  - delta_safe_rate
  - full_rebuild_rate
  - top fallback reasons
  - p50/p95 build_context_ms
  - p50/p95 sent_tokens
  - provider error rate split by mode
- Proceed to Phase 2 only if metrics are stable and no abnormal regressions are observed.

**Numeric gate thresholds (dev/synthetic acceptance, 2026-04):**

- `tail_delta_safe_pct >= 85%` on dedicated tail-only chats (e.g. `cache-delta-safe:*`).
- `provider_error_pct <= 1.0%` on gate window.
- `p95(build_context_ms) <= 2500`.
- `p95(sent_tokens) <= 120000` (guardrail against accidental runaway context).
- `rows >= 100` in gate window/slice (или больше для финального sign-off).

Automation:
- `mcp-tools/scripts/phase1_gate_report.py` — формирует PASS/FAIL по этим порогам.
- Рекомендуемый запуск:
  - `python mcp-tools/scripts/phase1_gate_report.py --hours 72 --tail-chat-ids 774,776 --out .tmp/phase1_gate_report.json`

Это не отменяет Step 2, но для ожидания ненулевого **`DELTA_SAFE`** в агрегате может понадобиться стабильнее меняющийся индекс, непрерывность сессии или отчёт с фильтром по одному длинному чату.

### Live chats: what you actually do (operator workload)

Telemetry пишется **автоматически** на каждый вызов LLM (`context_cache_metrics`). От вас не требуется отдельно «кормить» таблицу руками.

**Минимальный органический сценарий (рекомендуется сначала):**

- **Время:** 24–48 часов **календарных** с включённым Core и обычной работой в Colloquium (не 48 часов непрерывного чата).
- **Содержание:** несколько **разных** живых чатов, которые вы и так ведёте (разные проекты/темы/модели по возможности) — это и есть «mixed chats».
- **Объём сообщений:** явного порога в документе нет; ориентир — чтобы за окно набралось **сотни** интеракций суммарно (или сопоставимо с вашим порогом Phase 1), иначе p95/доли причин будут шумными. Если трафика мало, окно **дольше** или досбор сценариями ниже.
- **Действия оператора:** периодически смотреть отчёт (`agent/scripts/context_cache_report.py` или SQL по таблице), убедиться, что нет аномалий; **отдельный чек-лист «на каждое сообщение» не нужен**.

**Дополнительно (если мало живого трафика или нужны редкие причины инвалидации):**

- Прогоны из раздела **Tier B / Concrete Auto-Test Protocol** (эфемерные чаты с префиксом `cache-phase1-test:` и т.д.) — это **автоматизируемая** страховка, не замена живым чатам, а дополнение.
- **Быстро растущий контекст в одном чате** (последовательные реальные файлы `@attached_file#`, ответ модели про видимые строки и число постов): `mcp-tools/scripts/context_window_growth_runner.py` — описание чата `cache-context-growth:…` (подходит под `cleanup_runner_chats.py`). Опционально `--queries-per-file 2|3`: несколько разных задач подряд на тот же файл (строки / комментарии / код), чтобы чаще держать один и тот же индекс между соседними LLM-вызовами. **Автопроверка сдвига окна и FULL:** `--until-window-shift` опрашивает `context_cache_metrics` (нужны записи в таблице, не отключать `CQDS_CONTEXT_CACHE_METRICS_WRITE`); при успехе `status=ok_window_shift`, при исчерпании шагов — `timeout_no_window_shift` и код выхода 2. Пример: `--until-window-shift --max-total-turns 0 --max-files 120 --min-lines 220 --queries-per-file 2 --max-llm-steps 350 --out .tmp/context_shift.json`.
- После массовых прогонов — очистка эфемерных чатов скриптом `cleanup_runner_chats.py`.

**Фоновый запуск через `cq_process_ctl` (Colloquium MCP):**

- Инструмент **поддерживает оба режима**: по умолчанию subprocess идёт в **mcp-sandbox** (`host: false`), либо на **хост MCP** (`host: true`). Для длинных HTTP-раннеров к Core **не обязательно** упираться в sandbox.
- **`host: false`:** изолированная среда; до сервиса `colloquium-core` в docker-сети и до **`localhost:8008` хоста** доступа обычно **нет** → типичные `ConnectError` / таймауты при вызовах API.
- **`host: true`:** процесс выполняется там же, где MCP (часто ваш dev-хост с проброшенным портом **8008**). Имеет смысл для `context_window_growth_runner`, filewalk и любых скриптов с `COLLOQUIUM_URL=http://localhost:8008`. Управление: `spawn` → `process_guid`, затем `status` / `io` / `wait`, при необходимости `kill`.
- Альтернатива без MCP: локальный **Task Scheduler** / `Start-Process` / `nohup` с тем же `python` и переменными окружения.

**Стоит ли автоматизировать «живое»?**

- Сама **запись метрик** уже автоматическая.
- Имеет смысл автоматизировать **отчёт и алерт**: cron/Task Scheduler + `context_cache_report.py` (или экспорт в ваш мониторинг). Полностью автоматизировать «живое общение» не нужно — достаточно естественного использования в окне 24–48h плюс при необходимости ночных сценариев из протокола.

### When the product is dev-only (synthetic traffic only)

This is expected early in the project. Synthetic assistant tasks **do resemble Phase 1 soaks** (same telemetry, smaller N). The gap vs «mixed live chats» is not the label on traffic, but **coverage and realism**:

- **Volume:** you will have **orders of magnitude fewer** rows than a production window. That is acceptable if you **compensate with scenario diversity** and honest labeling: treat aggregates as a **synthetic baseline**, not as proof of production behavior.
- **What to optimize for instead of raw interaction count:**
  - **Scenario matrix** — deliberately exercise tail-only turns, follow-ups, simulated old-post edits, index/file attachment changes, and repeats without tail append (see Tier B / Concrete Auto-Test Protocol). Goal is to **hit each important `reason` and mode path**, not to duplicate the same cheap interaction hundreds of times.
  - **Stability over calendar time** — run small batches across **several days** (nightly or on-demand) so you still see drift/env issues, even if each day is low volume.
  - **Tests first for correctness** — Step 0 items (token trim, cache decision unit tests) matter **more** when you lack organic traffic; synthetic volume cannot replace missing regression tests.
- **Step 2 gate in dev:** keep the **same metrics** (rates, reasons, p95, errors), but relax the **interpretation**: define minimum **per-scenario** repetitions and acceptable error bars, and record in writing that Phase 2 enablement used a **synthetic acceptance** until real users arrive.

## Concrete Auto-Test Protocol (Core API + MCP Credentials)

Execution model:
- Test runner must use the same Core API endpoint and credentials already configured for `mcp-tools`
  (URL/username/password source identical to MCP runtime config).
- Runner creates dedicated ephemeral chats for test sessions and never reuses production discussion chats.
- Target project is selected from real DB projects (arbitrary project_id allowed by policy), then all tasks run in read-only analytical mode.

Mandatory behavior:
- Create a fresh chat per scenario run (`create_chat`), attach scenario tag in description.
- Select target project explicitly before analysis steps.
- Send only non-modifying prompts:
  - code review
  - architecture analysis
  - bug-risk assessment
  - dependency/flow explanation
- Explicitly forbid edit/write actions in prompt contract for test actor:
  - no patch/apply/edit requests
  - no migration/execution with side effects
- Keep dialogues short (e.g., 2-4 turns) to simulate interactive loop `LLM -> agent -> LLM` with bounded cost.

Recommended scenario template (read-only):
1) `list_projects` / choose project_id from DB.
2) `create_chat` with title prefix `cache-phase1-test:<scenario>:<timestamp>`.
3) Turn 1: ask model for focused analysis of a real module (no changes).
4) Turn 2: ask clarifying follow-up grounded in previous answer.
5) Optional Turn 3: ask risk summary or test-plan suggestions (still no edits).
6) Collect telemetry row and close scenario.

Protocol invariants:
- No write endpoints except creating chat messages themselves.
- No code/file mutation tools are invoked.
- Prompt text includes guard clause:
  - "Analysis only. Do not propose or execute modifications; answer in review mode."
- Each scenario records:
  - project_id, chat_id, actor/model
  - scenario_id, turn_count
  - cache mode/reason timeline
  - timing/tokens/error outcomes

## MCP-Driven Manual/Automated Runner Mode

The same protocol can be executed directly through MCP tools (operator run or scripted harness):
- `cq_project_ctl` for project list/select.
- `cq_chat_ctl` for create/send/wait/history.
- Optional batching via `requests[]` where safe.

Minimal MCP sequence (single scenario):
1) `cq_project_ctl` action=`list_projects`
2) `cq_chat_ctl` action=`create_chat` (scenario label)
3) `cq_chat_ctl` action=`send_message` (turn 1, read-only analysis prompt)
4) `cq_chat_ctl` action=`wait_reply`
5) `cq_chat_ctl` action=`send_message` (turn 2 follow-up)
6) `cq_chat_ctl` action=`wait_reply`
7) `cq_chat_ctl` action=`get_history` + telemetry extraction

Batch-friendly example:
- Combine independent chat-control steps in ordered `requests[]` when waiting boundaries are respected.
- Keep `wait_reply` as synchronization barriers between turns.

Acceptance criteria for this protocol:
- [ ] Runner can execute N scenarios end-to-end without manual intervention.
- [ ] All scenarios stay read-only by prompt and tool policy.
- [ ] Telemetry rows are persisted for every turn and can be aggregated daily.
- [ ] Failed scenario does not block suite; error is recorded with reason code.
